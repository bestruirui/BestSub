## 一、文档目的

本文档用于指导 BestSub 的二次开发，目标是：

1. 尽量复用上游现有能力
2. 尽量减少对核心流程的侵入
3. 先完成“家宽节点打标 + 前端筛选”
4. 再逐步扩展到“多 IP 质量源 / 多 API 评分”

本文档**不写具体代码实现**，只说明：

- 为什么这么设计
- 应该改哪些位置
- 每一阶段做什么
- 哪些内容暂时不做

## 二、当前项目现有能力梳理

BestSub 现有能力已经覆盖了本需求的大部分骨架，不需要重新造一套系统。

### 2.1 现有主流程

当前主流程可以概括为：

```text
订阅抓取
  -> 转换为 mihomo 节点
  -> 协议过滤
  -> 节点去重
  -> 存活检测
  -> 合并进 Node Pool
  -> 检测任务（Alive / Country / TikTok / Speed）
  -> 分享页按过滤条件导出订阅
```

### 2.2 本项目已经可以直接复用的能力

本次开发优先复用以下能力：

- 订阅抓取与节点收集
- Node Pool 节点池
- Check 检测任务体系
- Cron 定时调度
- 前端 Check 页面动态展示检测器
- 分享页按 `alive_status` 过滤节点

### 2.3 这意味着什么

这意味着本需求不应该重新设计为：

- 中间插队服务
- 独立常驻 addon service
- 独立外部进程回写

而应该优先设计为：

- **收集完成后的后处理检测任务**
- **作为新的 checker 接入现有 Check 体系**

---

## 三、最终设计决策

## 3.1 采用方案

### v1 最终采用

- [ ] 采用 **方案 B：独立的后处理 checker 模式**
- [ ] 不采用“中间插队服务”
- [ ] 不采用“外部独立服务”

### 设计解释

家宽判定不应插入抓取热路径，也不应绑定在 `node.Add()` 或 `main.go` 启动流程里。

更合适的方式是：

```text
节点先照常进入 Pool
  -> 后续由 residential checker 扫描 Pool
  -> 通过 IP 检测 API 给节点打标
  -> 前端分享页按标签筛选
```

这样做的原因：

1. 与上游主流程耦合最小
2. 后续跟上游同步时代码冲突最少
3. 可直接复用 Check/Cron/UI/日志能力
4. 家宽判定失败不会拖慢订阅抓取和节点入池

## 五、状态模型设计

## 5.1 设计原则

家宽标签最终必须能进入 BestSub 现有筛选体系，因此需要在节点状态中保留最小必要信息。

目标：

- 前端能筛选“家宽”
- 后端能区分“未检测”和“已检测非家宽”
- 尽量少改现有数据结构

## 5.2 状态位设计

继续复用 `AliveStatus` bit flag，并新增两个状态位：

| 状态 | 值 | 含义 |
|---|---:|---|
| `Alive` | `1 << 0` = `1` | 节点存活 |
| `Country` | `1 << 1` = `2` | 已获取国家 |
| `TikTok` | `1 << 2` = `4` | 可访问 TikTok |
| `TikTokIDC` | `1 << 3` = `8` | TikTok IDC |
| `Residential` | `1 << 4` = `16` | 已判定为家宽 |
| `ResidentialChecked` | `1 << 5` = `32` | 已完成家宽判定 |

## 5.3 三种状态语义

| Residential | ResidentialChecked | 语义 |
|---|---|---|
| 0 | 0 | 未检测 / 检测失败 / 待重试 |
| 0 | 1 | 已检测，确认非家宽 |
| 1 | 1 | 已检测，确认家宽 |

约束：

- 不允许出现 `Residential=1` 且 `ResidentialChecked=0`
- 只有在外部数据源返回“可判定结果”时，才写入 `ResidentialChecked`
- 如果请求超时、失败、限频、结果异常，则保持未检测状态

## 六、v1 架构设计

## 6.1 总体流程

```text
订阅抓取
  -> 节点入 Pool
  -> 用户手动运行 / 定时运行 residential checker
  -> checker 扫描 Pool 中目标节点
  -> 调用 IP 评估 API
  -> 回写 Residential / ResidentialChecked
  -> 分享页筛选出家宽节点
```

## 6.2 v1 的关键设计原则

### 原则 1：不侵入主抓取热路径

不把家宽检测塞进：

- 订阅抓取
- `node.Add()`
- `mergeNodesToPool()`

这些位置是上游核心流程，改动它们会增加冲突风险。

### 原则 2：把家宽判定视为新的检测器

Residential 应作为一个新的 checker，接入现有：

- Check 注册机制
- Check 页面
- Cron 调度
- 手动运行入口
- 日志体系

### 原则 3：先完成单一职责

v1 只做：

- 节点打标
- 前端筛选

不做：

- 复杂 provider 管理
- 多源评分系统
- 动态规则引擎

## 7.3 分享筛选行为

v1 目标不是“仅有家宽标签”，而是尽量筛出：

- 家宽
- 且当前可用

因此推荐筛选语义为：

- `Alive + Residential`

而不是只筛 `Residential`。

这意味着产品层面需要明确：

1. “家宽节点”是标签语义
2. “完全可用的家宽节点”是组合筛选语义

## 八、v1 检测器设计思路

## 8.1 Residential checker 的职责边界

Residential checker 只负责：

- 遍历候选节点
- 调用外部 IP 检测 API
- 将结果翻译为本地状态位

它不负责：

- 订阅抓取
- 节点入池
- 节点分享导出
- 独立服务常驻运行

## 8.2 v1 数据源策略

v1 先使用单个公开 API 或单个固定 provider 跑通流程。

原因：

1. 先验证整个打标和筛选链路
2. 降低初期复杂度
3. 避免一开始就陷入 provider 管理和规则引擎设计

## 8.3 v1 判定思路

v1 采用“可用优先、保守写入”的原则：

- 可明确判定为机房：写入已检测、非家宽
- 可明确判定为住宅：写入已检测、家宽
- 结果不稳定或无法判定：不写已检测，留待后续再扫

为避免开发范围失控，以下事项明确延后：

## 11.1 Phase 2：多 Provider 聚合（v1 稳定后）

状态：- [ ] 未开始

目标：

- 在保持“checker 后处理模式”不变的前提下
- 从单一 provider 扩展到多个 IP 质量源
- 让家宽判定更稳

### 计划内容

- [ ] 在 residential checker 内部引入 provider 抽象
- [ ] 支持多个 provider 顺序调用或聚合调用
- [ ] 统一 provider 返回结果结构
- [ ] 设计最小聚合策略

### 实现思路

这一阶段仍然不改变大架构，依旧保持：

- 节点先入 Pool
- checker 后处理
- 前端按状态位筛选

变化只发生在 residential checker 内部。

### 聚合策略建议

优先使用保守聚合策略：

1. 任一 provider 明确判机房 -> 非家宽
2. 没有机房结论，且至少一个 provider 明确判住宅 -> 家宽
3. 所有 provider 都不可判定 -> 保持未检测

### 编码位置建议

- `internal/core/check/checker/residential.go`
- 或从中继续拆出 provider 相关文件

但仍应保持：

- checker 是对外入口
- provider 只是内部模块

## 十三、最终结论

本项目的家宽筛选功能，最终采用如下开发路线：

### 第一阶段

- [ ] 采用 **后处理 checker 模式**
- [ ] 不改抓取主流程
- [ ] 不改启动主流程
- [ ] 不做独立外部服务
- [ ] 不做独立配置平台

### 第二阶段以后

- [ ] 在 residential checker 内逐步加入多 Provider 聚合
- [ ] 再逐步加入配置化能力
- [ ] 最后视实际需求决定是否做评分与规则系统

### 这条路线的优点

1. 侵入低
2. 可快速落地
3. 易于跟上游同步
4. 风险分阶段收敛
5. 不会在 v1 就陷入复杂平台开发
